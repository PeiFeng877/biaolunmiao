## 1. 文档目标
本文件是辩论喵主仓协作宪章。它不是“说明文”，而是可执行规则集，目标是让 AI 与开发者在同一标准下高效协作，并保证代码与文档长期同构。

## 2. 核心原则
1. 文档即系统语义层，代码即系统执行层，两者必须同步演进。
2. 先契约后实现，先边界后细节，先可验证再扩展。
3. 优先消灭特殊分支，而不是堆叠条件判断。
4. 所有跨端协作以单一事实源（SSOT）为准，禁止多处“各自为政”。

## 3. 语言与命名规范
1. 与用户沟通统一使用中文。
2. 项目文档统一使用中文。
3. 思考与代码实现使用英文。
4. 新增文档文件名统一使用中文，例外仅 `agents.md`。

## 4. 工作目录与模块边界
1. 根目录: `./`
2. iOS 工程: `./bianlunmiao-ios/`
3. 后端工程: `./辩论喵-后端/`
4. 跨端文档中心: `./docs/`

跨端公共规范放 `./docs/`，端内实现细节放各子项目目录，避免重复维护。
当前仓库为单仓模式，统一跟踪 `docs/`、`bianlunmiao-ios/`、`辩论喵-后端/` 与治理脚本。

## 5. 分形文档系统（GEB）
### 5.1 三层结构
1. L1: 根 `agents.md`，定义全局协作规则、边界与质量门槛。
2. L2: 子目录 `agents.md`，定义模块职责、文件清单与局部约束。
3. L3: 文件级 `[PROTOCOL]` 头，保证关键文件具备可追踪的语义入口。

### 5.2 同构约束
1. 代码结构变化必须在对应层级文档显式体现。
2. 文档声明的结构与职责必须能在代码中找到真实映射。
3. 任一层过时都视为任务未完成。

### 5.3 触发更新
1. 新增/删除/重命名文件或目录。
2. 模块职责变化或依赖关系变化。
3. API、数据模型、测试门禁、发布流程发生变化。

### 5.4 固定协议头
所有 L2/L3 文档应包含：
`[PROTOCOL]: 变更时更新此头部，然后检查 agents.md`

## 6. 代码品位准则
### 6.1 Good Taste
1. 优先通过重构数据结构或抽象边界来消除特殊情况。
2. 三个以上同类分支是重构信号，不是“继续加 if”的理由。
3. 让正常路径覆盖边界路径，减少例外处理。

### 6.2 Pragmatism
1. 先交付最小可运行方案，再做增强。
2. 不为假想复杂度提前过度设计。
3. 每个技术决策都要能回答“当前收益是什么”。

### 6.3 Simplicity
1. 函数应短小、单一职责、命名直白。
2. 控制嵌套深度，超过三层应优先拆分。
3. 优先降低耦合与状态扇出，保持数据流可解释。

### 6.4 坏味道清单
1. 僵化: 小改动触发连锁修改。
2. 冗余: 同一逻辑重复实现。
3. 循环依赖: 模块互相缠绕。
4. 脆弱性: 一处变更破坏无关功能。
5. 晦涩性: 代码意图不清。
6. 数据泥团: 应聚合的数据长期分散。
7. 不必要复杂: 抽象层级多于业务需要。

发现坏味道时，默认给出可执行优化方案，不以“后续再说”结束。

## 8. 标准执行流程（AI/人工统一）
1. 澄清目标与影响范围。
2. 锁定将要修改的路径与禁止路径。
3. 先更新契约或规范文档，再进入代码实现。
4. 实现后执行最小验证（lint/test/脚本检查）。
5. 回写文档与变更日志，确保同构。
6. 输出可审查结果：改动点、验证结果、剩余风险。

## 9. 工程与质量门禁
### 9.1 结构约束
1. 建议单文件不超过 800 行。
2. 建议单层目录文件数不超过 8 个，超限需拆分。

### 9.2 iOS 基线
1. 治理脚本: `swift docs/03_Governance/tools/governance_audit.swift --mode check`
2. 按项目策略执行 lint 与测试门禁。

### 9.3 后端基线
1. `make lint`
2. `make test`

### 9.4 文档基线
1. 必须包含 `[PROTOCOL]`、版本、日期、变更日志。
2. 目录变更必须同步更新相应 `agents.md`。

## 10. Git 与提交流程
1. 分支命名: `codex/<topic>`
2. 提交格式: `feat|fix|docs|refactor: <scope> <summary>`
3. 跨端接口变更必须同步更新 `docs/03_接口与数据契约/`
4. 禁止“只改代码不改文档”的孤立提交
5. 单仓模式下统一跟踪 `docs/`、`bianlunmiao-ios/`、`辩论喵-后端/`、`scripts/` 与治理文件
6. 同一功能可在一个提交中同时改动文档、iOS 与后端，但仍应按功能主题拆分提交

### 10.1 提交范围判定（强制）
1. iOS 改动（`bianlunmiao-ios/**`）可与同一功能所需的 `docs/**` 一并提交。
2. 后端改动（`辩论喵-后端/**`）可与同一功能所需的 `docs/**` 一并提交。
3. 同一功能同时涉及 iOS、后端、文档时，允许在一个提交中落地；若是不同主题改动，必须拆分提交。
4. 禁止把无关治理改动、产品文案改动和业务实现混入同一提交。

### 10.2 提交前自检（强制）
1. 执行 `pwd`，确认当前目录位于单仓工作区。
2. 执行 `git rev-parse --show-toplevel`，确认 Git 根路径为主仓。
3. 执行 `git status --short`，确认待提交文件与本次功能范围一致。
4. 若提交同时包含 `docs/`、`bianlunmiao-ios/`、`辩论喵-后端/`，提交说明中必须写清业务原因。

## 11. 禁止行为
1. 改代码不检查文档同构。
2. 删/改结构但不更新对应层级 `agents.md`。
3. 未定义契约就推进跨端联调。
4. 绕过门禁直接合并或发布。

## 12. 完成定义（DoD）
1. 目标范围明确，影响路径可解释。
2. 代码实现可运行。
3. 最小测试或检查通过。
4. 文档与代码完成同构更新。
5. 风险、回滚方式、后续动作可说明。

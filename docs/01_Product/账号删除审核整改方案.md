## 1. 背景与目标
- 本次 App Review 相比上轮已经前进一大步，模板代码问题已解决。
- 当前剩余阻塞项是账号体系已支持登录/创建，但 App 内缺少“发起账号删除”的功能入口。
- 本文档统一沉淀审核拒信、中文翻译、整改方案、实现边界与提审回复口径，作为本次整改的单一事实源。

## 2. 审核拒信原文
### 2.1 基本信息
- Submission ID: `da2f512b-9d9c-4294-be62-41989de8ae14`
- Review date: `2026-03-05`
- Review Device: `iPhone 17 Pro Max`
- Version reviewed: `1.0.0`

### 2.2 原文摘录
> Guideline 5.1.1(v) - Data Collection and Storage
>
> The app supports account creation but does not include an option to initiate account deletion.
>
> Apps that support account creation must also offer account deletion to give users more control of the data they've shared while using an app.
>
> Only offering to temporarily deactivate or disable an account is insufficient.
>
> If users need to visit a website to finish deleting their account, include a link directly to the website page where they can complete the process.

## 3. 拒信中文翻译
### 3.1 问题描述
- 应用支持账号创建，但没有提供用户主动发起账号删除的入口。
- 只提供“临时停用”或“禁用账号”是不够的。
- 如果删除流程需要跳转到网页完成，应用内必须直接提供能抵达该页面的链接。
- 除非属于强监管行业，否则不能要求用户必须通过客服、电话或邮件才能完成账号删除。

### 3.2 Apple 的明确要求
- 删除入口必须由用户自己在产品内触发。
- 可以做确认弹窗，避免误删。
- 但不能把删除流程做成“只是退出登录”“只是停用”“只是联系客服”。

## 4. 合规判断与结论
### 4.1 本次拒绝的真实缺口
- 不是“没有登录功能”，而是“有登录功能却没有删除账号功能”。
- 当前 `ProfileMoreView` 已经有“退出登录”，但“退出登录”不等于“删除账号”。

### 4.2 结论
- 必须新增独立的“删除账号”功能入口。
- 删除动作必须在 App 内完成闭环，不能只提供说明文案。
- 后端可以不做物理删除，但对外语义必须是“账号已删除”，而不是“暂时停用”。

## 5. 方案选项评估
### 5.1 选项 A：30 天不登录才真正删除，期间登录可恢复
- 描述: 用户点击删除后进入“待删除”状态，30 天内重新登录则恢复账号。
- 问题: 从用户视角，这更接近“停用/冷静期”，容易被 Apple 认定为并未真正提供账号删除。
- 结论: 不推荐作为本次审核整改方案。

### 5.2 选项 B：用户确认后立即进入“已删除”状态，后端仅做软删除
- 描述: 用户在 App 内确认删除后，服务端立即将账号标记为 `deleted`，保留必要数据用于审计或风控，不再视为活跃账号。
- 优点: 用户侧语义清晰，最符合 Apple “account deletion” 要求，实现成本也最低。
- 结论: 推荐采用。

## 6. 推荐方案（采用）
### 6.1 用户流程
1. 用户进入“我的 > 更多”页面。
2. 在“退出登录”下方新增“删除账号”危险操作项。
3. 第一次确认弹窗说明后果:
   - 删除后将立即注销当前账号。
   - 账号状态将变更为“已删除”。
   - 与该账号关联的业务数据将不再继续在产品内使用。
   - 出于合规、审计或风控需要，服务端可保留最小必要记录，但账号对用户视角应视为已删除。
4. 用户确认后进入第二次强确认:
   - 按钮文案使用“确认删除账号”。
   - 明确提示“该操作不可撤销”。
5. 调用后端删除账号接口成功后:
   - 本地清空登录态。
   - 跳回登录页。
   - Toast 或结果页提示“账号已删除”。

### 6.2 文案建议
- 入口文案: `删除账号`
- 一级确认标题: `确认删除账号？`
- 一级确认说明:
  - `删除后你将退出当前账号，账号状态会立即变更为已删除。`
  - `为满足法律法规、审计或风控要求，服务端可能保留最小必要记录，但该账号将不再作为可登录、可继续使用的有效账号。`
- 二级确认标题: `该操作不可撤销`
- 二级确认说明:
  - `确认后将立即删除当前账号。`
- 成功提示: `账号已删除`

### 6.3 关于“30 天提醒”的处理建议
- 不建议把“30 天内重新登录可恢复”作为主方案文案。
- 如果业务上确实需要保留数据清理窗口，可以仅作为服务端内部保留策略，不对用户承诺“登录即可恢复”。
- 如需展示，可写成:
  - `删除申请提交后，相关数据将在服务端清理策略下处理。`
- 原因: 一旦明确承诺“30 天内登录可恢复”，就会削弱“账号删除已完成”的产品语义。

## 7. 后端语义与接口约束
### 7.1 推荐数据语义
- 新增账号状态: `deleted`
- 记录删除时间: `deletedAt`
- 可选记录删除来源: `deletionSource = ios_app_review_fix`

### 7.2 行为约束
- 账号一旦标记为 `deleted`，原有登录凭证不应继续进入正常产品态。
- 使用同一 Apple 账号再次登录时，后端需要有明确策略，二选一即可:
  - 策略 A: 直接阻止登录，并返回“账号已删除”的错误码。
  - 策略 B: 允许按“新账号”重新注册，但必须与旧账号数据彻底隔离。
- 本次整改更稳妥的建议: 先采用策略 A，避免“删除后又恢复旧账号”的语义歧义。

### 7.3 接口建议
- `DELETE /v1/account`
- 或 `POST /v1/account/deletion`
- 返回字段建议:
  - `status = deleted`
  - `deletedAt`
  - `message`

## 8. iOS 实现范围
### 8.1 页面与交互
- 修改 `ProfileMoreView`
- 新增“删除账号”入口
- 新增两段式确认交互
- 成功后清空会话并返回登录页

### 8.2 ViewModel 与 Store
- `ProfileSettingsViewModel` 新增删除账号动作
- `AppStore` 新增账号删除请求与成功后的会话清理
- 需要区分“退出登录”和“删除账号”两个动作，避免复用造成语义混淆

### 8.3 测试建议
- 单元测试:
  - 删除账号成功后会话被清空
  - 删除账号失败时保留当前登录态并展示错误
- UI 测试:
  - “更多”页能看到“删除账号”
  - 两次确认后触发删除
  - 删除成功后回到登录页

## 9. 提审回复建议
### 9.1 英文回复草案
> We have added an in-app account deletion entry in `My > More > Delete Account`.
>
> Users can now initiate and confirm account deletion directly inside the app without contacting customer service or visiting an external website.
>
> After confirmation, the account is marked as deleted on the server and the user is signed out immediately.

### 9.2 中文留档
- 我们已在应用内新增账号删除入口，位置为 `我的 > 更多 > 删除账号`。
- 用户可以直接在 App 内完成删除确认，无需联系客服，也无需跳转外部网页。
- 用户确认后，服务端会立即将账号标记为“已删除”，并同步退出当前登录态。

## 10. 开发前待确认事项
- 是否接受“软删除即对外删除”的语义定义。
- 删除后的再次登录策略，是否采用“禁止恢复旧账号”。
- 是否需要在隐私政策中补充“账号删除与最小必要留存”表述。
- 后端接口是直接新增删除接口，还是在现有用户接口上扩展。

2. 与后端对齐账号状态、错误码和再次登录策略。
3. 完成 iOS 端入口、确认弹窗、删除请求和结果处理。

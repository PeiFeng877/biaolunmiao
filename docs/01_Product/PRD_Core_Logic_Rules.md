# 辩论喵 (BianLunMiao) - MVP 1.0 PRD（队内日程同步）

[PROTOCOL]: 变更时更新此头部，然后检查 GEMINI.md

**版本**: v1.1  
**日期**: 2026-02-04  
**平台**: iOS App  
**定位**: 队伍内的「赛程分配 + 冲突检测 + 日历同步」工具  
**目标**: 让队伍的比赛安排不再依赖人肉记忆与微信群转述。

---

## 1. 范围与非目标

### 1.1 本版本解决什么（MVP 1.0）
- 队伍创建/加入/审批，形成可管理的队内成员关系。
- 队伍视角创建赛程：指派我方上场人员、录入对方信息、标注时间/地点。
- 赛程写入队伍内时间线，并同步到个人手机系统日历（可开关）。
- 队伍内冲突检测：管理员在指派人员时能看到站内时间冲突。
- 战绩沉淀（轻量）：赛果、最佳辩手等，进入个人/队伍的最近记录。

### 1.2 本版本不做什么（明确不承诺）
- 不做“赛事方统一发布赛程、全网报名参赛”的平台模式。
- 不做跨队伍的公共赛事目录、赛制编排器、晋级表等。
- 不做复杂评分/技术统计；只记录最小结果字段。
- 不做外部日历的完整事件读取；仅保留未来扩展空间（忙闲块可选）。

## 2. 核心角色与权限

### 2.1 角色
- **队长 (Owner)**：队伍创建者。
- **管理员 (Admin)**：由队长指派。
- **队员 (Member)**：普通成员。

### 2.2 权限矩阵（MVP 1.0）

| 操作 | 队长 | 管理员 | 队员 |
| :--- | :---: | :---: | :---: |
| 创建队伍 | ✅ | ✅ | ✅ |
| 编辑队伍资料 | ✅ | ✅ | ❌ |
| 审批入队申请 | ✅ | ✅ | ❌ |
| 邀请成员/移除成员 | ✅ | ✅ | ❌ |
| 创建赛程 | ✅ | ✅ | ❌ |
| 指派上场人员 | ✅ | ✅ | ❌ |
| 录入/修改赛果 | ✅ | ✅ | ❌ |
| 查看队伍赛程与战绩 | ✅ | ✅ | ✅ |

## 3. 信息架构（3 个 Tab）

### 3.1 队伍页（Tab 1）
- **队伍列表**：展示我加入过/当前所在的队伍，支持切换“当前队伍”视角。
- **创建/加入队伍**：
  - 创建：填写队名、简介、头像（可选）。
  - 加入：通过队伍码/搜索发起申请，等待审批。
- **身份状态**：展示我在每个队伍的角色（队长/管理员/队员）。
- **队伍详情**：
  - 成员列表（按角色分组或标签）。
  - 待处理申请（仅队长/管理员可见）。
- **队员详情**：
  - 基本资料（头像、昵称等）。
  - 个人赛程：过去比赛记录 + 未来已安排赛程。
  - 个人战绩（由赛果沉淀，MVP 1.0 先做最近 N 场/胜负）。

### 3.2 赛事页（Tab 2，以“当前队伍”为视角）
- **赛事/赛程查看（全员可见）**：
  - 即将到来（未来赛程）
  - 已结束（历史记录）
- **赛事/赛程管理（仅队长/管理员）**：
  - 创建赛程（MVP 1.0 允许“手动录入外部赛事信息”，不等于主办方办赛）
  - 指派我方上场人员（甲乙丙丁等 + 辩位）
  - 录入对方信息（MVP 1.0 先支持手动文本/占位身份）
  - 录入赛果（胜负/比分可选/最佳辩手等）

### 3.3 我的/设置页（Tab 3）
- **个人资料**：头像、昵称等编辑。
- **日历同步设置**：
  - 同步开关（是否写入系统日历）
  - 同步策略（默认“需确认后同步” / 可选“自动同步”）
- **通知设置**：赛程变更、指派、赛果更新等。

## 4. 核心交互闭环（MVP 1.0）

### 4.1 队伍闭环：建队 -> 入队 -> 成员可管理
1. 管理员注册登录，创建队伍。
2. 其他人注册后申请加入队伍。
3. 队长/管理员审批通过，成员进入队伍。

### 4.2 赛程闭环：创建 -> 指派 -> 同步 -> 结算
1. 队长/管理员创建一条赛程，填写：
   - 时间（开始/结束或开始+时长）
   - 地点（线下地址/线上链接）
   - 对阵信息（对方队伍名；对方选手信息 MVP 1.0 允许为空或文本）
2. 指派我方上场人员（可标记辩位）。
3. 系统动作（同步）：
   - 被指派队员在 App 内收到赛程卡片（含时间/地点/对阵/辩位）。
   - 若用户开启日历同步：一键写入/更新系统日历事件（变更时同步更新）。
4. 比赛结束后，队长/管理员录入赛果：
   - 胜负（必填）
   - 比分（选填）
   - 最佳辩手（选填）
5. 沉淀展示：
   - 队伍最近比赛记录更新
   - 被指派队员个人履历/战绩更新

### 4.3 冲突检测（先做站内冲突）
- 触发时机：管理员在“指派上场人员”步骤中选择队员时。
- 冲突定义（MVP 1.0）：
  - 同一用户在同一时间段内存在其它赛程（任意队伍）即提示冲突。
- 展示方式：
  - 人员列表右侧标红/提示“与 XX 赛程冲突”，点击可展开冲突详情。
- 处理：
  - MVP 1.0 仅提示，不强制阻止提交（允许管理员“仍然指派”）。

## 5. 数据口径（产品层，不含实现细节）

### 5.1 关键对象
- **用户 (User)**：登录身份与个人资料。
- **队伍 (Team)**：成员集合。
- **队伍成员关系 (Membership)**：用户在队伍内的角色（队长/管理员/队员）。
- **赛程 (ScheduleItem / Match)**：一场比赛（时间、地点、对阵、我方阵容、结果）。
- **对手 (Opponent)**：MVP 1.0 先用“文本 + 可选头像/备注”承载；后续可升级为可认领身份。

### 5.2 状态（建议）
- 赛程状态：`Draft(草稿)` -> `Scheduled(已发布)` -> `Completed(已结束)`
- 同步状态（个人维度）：`NotSynced(未同步)` / `Synced(已同步)` / `Outdated(需更新)`

---

## 6. 成功指标（MVP 1.0）
- 创建队伍后 7 天内，队伍产生 ≥1 条赛程的比例。
- 赛程创建后，被指派队员完成日历同步的比例。
- 冲突提示被点击查看的比例（验证需求真实存在）。
- 队伍在 30 天内的留存（以“持续创建赛程/录入赛果”作为活跃指标）。

## 7. 依赖与开放问题（需要尽早定）
- 日历同步的默认策略：默认“需确认后同步”还是“自动同步”？
- 对方选手信息的最小字段：仅队名即可？是否必须填对手选手？
- 赛程的最小字段：时间、地点、对阵、我方阵容哪些必填？
- 冲突检测是否需要考虑“准备/路程缓冲时间”（MVP 1.0 可不做，但要留开关口径）。

# FC 单正式环境与本地开发方案

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v2.1
**日期**: 2026-03-22
**适用范围**: `bianlunmiao-ios/`、`辩论喵-后端/`、`docs/04_测试与发布/` 的现行联调、发布与资源收口

## 1. 目标
1. 将现行后端收敛为 `本地开发 + 单正式环境` 两态，不再保留云端测试服。
2. 正式后端统一采用 `FC Web 函数 + FastAPI`，先使用 FC 默认域名，后续再按需要切换正式自定义域名。
3. 数据库不复用旧实例，改为新购杭州 `RDS PostgreSQL Serverless`，开启 `AutoPause`，按购买页可选的最小配置下单。
4. 正式存储继续使用现有 `OSS`；Web 管理后台纳入本轮正式发布范围，并通过正式 `Vercel prod` 接入同一后端入口。
5. `TestFlight` 与正式后端共用同一套服务与数据库，但必须使用专用测试账号与专用命名规范。

## 2. 运行态决策
1. 现行目标架构为 `FastAPI + FC + RDS PostgreSQL Serverless + OSS`。
2. 本阶段只保留 `local` 与 `prod` 两态，不再定义 `stg`、双环境或云端预发服。
3. 现有 `EMAS`、`SAE`、`CLB`、`ECS/Caddy` 与旧 `RDS` 文档仅作为历史参考，不再作为新方案执行依据；相关旧源码只允许保留在本机 `./.data/local-archive/` 归档，不再纳入 Git 主路径。

## 3. 环境划分
| 环境 | Service Space | 目标 | 默认入口 |
| --- | --- | --- | --- |
| `local` | 本机 | 开发、回放、最小烟测 | `http://127.0.0.1:8788` |
| `prod` | FC 默认域名 | 正式流量、发布演练、TestFlight 验收 | FC 部署产物生成的默认访问域名 |

### 3.1 当前已确认的 `prod` 资源
1. 入口：FC 默认域名，先承接正式流量与 TestFlight 验收。
2. 后端运行时：现行目标为 `FastAPI`，由 FC Web 函数承载。
3. 存储：正式图片与对象继续使用 `OSS`。
4. 数据库：新购杭州 `RDS PostgreSQL Serverless`，开启 `AutoPause`，不继承旧数据。
5. 域名：`api.bianlunmiao.top` 本轮暂不接入，等你后续通知再切。

## 4. 接口与函数拆分
1. 采用“单个公网业务入口 + 健康检查”的方式：`GET /healthz` 与 `POST /api`。
2. `healthz` 只承载健康检查；`api` 承载 iOS 的全部 RPC 动作分发。
3. 动作命名采用点式命名，如 `users.me.get`、`teams.create`、`tournaments.list`。
4. Web 管理后台本轮接入，浏览器侧统一通过 `/api/proxy/*` 代理到同一后端入口，不再定义额外线上链路。

## 5. 数据与存储
1. 数据层迁移到新购的杭州 `RDS PostgreSQL Serverless`，直接新建 schema，不做旧库数据迁移。
2. 现行业务表以 App 业务为中心，最小集合为：
   - `users`
   - `sessions`
   - `teams`
   - `team_members`
   - `join_requests`
   - `tournaments`
   - `matches`
   - `match_rosters`
   - `messages`
   - `schedule_sources`
   - `media_files`
   - `counters`
3. `TestFlight` 共享正式数据库，但只允许使用专用测试账号、专用队伍名和专用赛事命名规范。
4. 旧数据不迁移，正式切流前只允许通过新库种子和测试账号验证业务闭环。
5. 媒体上传继续使用 `OSS`，本地开发可用本地文件或 mock provider 代替。

## 6. 安全与开关
1. 本地开发允许 `debug token`；正式环境禁止 `debug token`。
2. 正式环境必须执行 Apple 校验，包含实时 JWKS、`issuer`、`audience`、`exp`、`sub` 与签名校验。
3. Web 管理后台使用独立管理员体系，现行通过 `admin.*` RPC 与 `admin_*` 数据模型承载，不复用 App 用户会话。
4. 正式部署参数由 FC 部署配置与环境变量承载，禁止把密钥写入仓库。

## 7. 多 Agent 拆分
1. `Agent A`: 端点契约、共享错误码、鉴权与请求校验。
2. `Agent B`: FC 默认域名、部署配置、密钥与发布口径。
3. `Agent C`: PostgreSQL schema、索引、唯一键、种子数据与存储语义。
4. `Agent D`: FastAPI 业务实现与共享业务模块。
5. `Agent E`: iOS 接入、本地联调与正式 smoke 验证、覆盖配置整理。
6. `Agent F`: 文档、测试门禁、发布治理与回滚说明。

## 8. 迁移顺序
1. 先冻结契约与数据模型，再完成本地 FastAPI 联调闭环。
2. 再把正式服务部署到 FC 默认域名，跑通健康检查、登录、队伍与赛事主链路。
3. 然后创建新购的杭州 `RDS PostgreSQL Serverless`，并完成 schema 与种子初始化。
4. 最后用 `TestFlight` 复用正式后端验证主流程，待你通知后再切正式自定义域名。

## 9. 验收标准
1. 本地 FastAPI 能启动，`GET /healthz` 与 `POST /api` 可用。
2. iOS 本地开发能完成登录、资料更新、队伍、赛事、场次与媒体上传的最小闭环。
3. FC 默认域名下能完成正式 smoke，且 `TestFlight` 可复用同一套后端与数据库。
4. 新购 RDS PostgreSQL Serverless 完成 `AutoPause` 开启、schema 初始化与最小种子写入。
5. 业务契约、错误码、数据库模型、发布流程与资源申请说明全部同步到 SSOT。

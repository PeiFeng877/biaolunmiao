**范围**: 基于 `BianLunMiao` 当前代码（`AppStore + ViewModels + Views`）整理。

## 1. 当前产品结构总览

当前 App 为 5 Tab 结构：
1. 队伍
2. 赛事
3. 日程
4. 消息
5. 我的

数据层现状：
- 所有业务数据集中在 `AppStore` 内存状态。
- 初始数据来自 `MockData`。
- 无真实后端请求、无持久化数据库、无用户登录态（当前默认固定 `currentUser`）。

---

## 2. 功能域与业务闭环

## 2.1 队伍域（Team）

已实现能力：
1. 创建队伍（名称/Slogan/头像）。
2. 队伍列表展示（我的队伍 + 可发现队伍）。
3. 队伍搜索（名称或公开 ID）。
4. 入队申请提交（按队伍 ID/搜索结果）。
5. 入队审批（通过/拒绝）。
6. 队伍管理：
- 编辑队伍资料。
- 成员移除。
- 管理员升降级。
- 队长移交。
- 队长解散队伍，成员退出队伍。
7. 成员详情查看（过往比赛、近期赛程）。

业务规则（已在代码中体现）：
1. 同一申请人对同一队伍，待审批申请不可重复提交。
2. 只有 Owner/Admin 可审批入队申请。
3. Owner 不能被移除，且只有 Owner 可移交队长。
4. 非 Owner 才可主动退队。

当前缺口：
1. “邀请成员”按钮目前是占位提示，未实现真实邀请流程。
2. 队伍能力完全本地，重启后不保真（除头像文件外）。

## 2.2 赛事与场次域（Tournament + Match + Roster + Result）

已实现能力：
1. 赛事列表（搜索、状态筛选、创建）。
2. 赛事详情查看与赛事信息编辑（名称/简介/状态）。
3. 场次管理：
- 新建/编辑场次。
- 设置开赛时间、地点、赛制、辩题、对手名称。
- 指定我方持方（正/反）。
4. 阵容指派（按赛制约束辩位人数，校验重复人/重复辩位）。
5. 场次状态流转（scheduled -> ready -> ongoing -> finished）。
6. 赛果录入（胜方、比分、结果备注、最佳辩手辩位）。

业务规则（已在代码中体现）：
1. 场次结束时间固定为开始时间 + 90 分钟。
2. 同一场次 A/B 队不可相同。
3. 阵容成员必须来自对应队伍成员。
4. 只有赛事创建者或可管理队伍的管理者可执行关键操作（按具体动作校验）。

当前缺口：
1. 赛事参与队伍来源仍依赖本地数据推断，缺少标准报名审核流程 API。
2. 没有淘汰赛对阵树/轮次引擎（当前偏“场次管理”）。

## 2.3 日程域（Schedule）

已实现能力：
1. 月历视图 + 日详情时间线切换。
2. 多来源聚合展示：
- 我
- 个人
- 队伍
- 赛事
3. 来源管理（添加、删除、启用/停用）。
4. 批量同步到系统日历（EventKit）。
5. 单条同步到系统日历。
6. 日历去重与更新（通过本地映射 + 标记字段）。

业务规则（已在代码中体现）：
1. `me` 来源不可删除且默认启用。
2. 同一来源不可重复添加。
3. 日历同步做幂等更新，避免重复写入。

当前缺口：
1. 来源配置目前存在 `UserDefaults`，不支持跨设备同步。
2. 无服务端冲突检测与提醒机制。

## 2.4 消息域（Message）

已实现能力：
1. 消息流聚合展示（入队申请 + 系统通知）。
2. 待审批申请详情查看。
3. 在消息详情内审批（通过/拒绝）。
4. 个人申请结果回流到消息流。

当前缺口：
1. 无真实推送链路（APNs/厂商推送均未接入）。
2. 系统消息当前由本地 Mock 驱动，不是后端事件驱动。

## 2.5 我的域（Profile）

已实现能力：
1. 个人资料展示（昵称、ID、头像）。
2. 编辑资料（昵称/头像）。
3. 已完成比赛时间线。
4. 用户协议与隐私政策静态页面。

当前缺口：
1. 用户协议/隐私政策为本地静态文本，未接入服务端版本化内容。
2. 资料修改未经过服务端校验链路。

## 3. 权限模型（现状）

团队权限：
1. `owner` > `admin` > `member`
2. 入队审批与成员管理要求 `owner/admin`。
3. 队长移交、解散属于 `owner` 特权。

赛事权限：
1. 赛事创建者可管理赛事信息和场次核心变更。
2. 可管理参赛队伍的用户可以维护本队阵容/部分场次信息。

---

## 4. 当前架构判断（对后端化的意义）

结论：
1. 产品交互骨架已完整，进入“数据后端化”阶段是合适时机。
2. 前端侧核心工作不是重画 UI，而是把 `AppStore` 现有写操作逐步替换为 API 调用。
3. 优先级应先打通：用户/队伍/申请，再到赛事/场次/阵容，最后消息推送和运营能力。

# 当前系统架构基线

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v1.0
**日期**: 2026-03-05
**适用范围**: `bianlunmiao-ios/`、`bianlunmiao-flutter/`、`辩论喵-后端/`

## 1. 当前系统形态
1. 架构模式：客户端 + 后端 API 的前后端分离模式。
2. 后端技术：FastAPI + PostgreSQL + Alembic。
3. 生产入口：`https://api.bianlunmiao.top/api/v1`。
4. 联调策略：`stg` 环境按需启停，`prod` 在提审窗口保持在线。

## 2. 端侧现状
1. iOS：为当前主交付端，已具备 `1.0.0 (4)` 提审能力并完成重新提审材料整理。
2. Flutter：处于迁移推进阶段，承担 Android 方向后续实现，不再延续独立 Android 原仓路线。
3. 根仓 `docs/`：作为跨端 SSOT，统一维护产品、架构、契约、测试与发布规范。

## 3. 后端现状
1. 鉴权：已支持 `POST /auth/apple` 主链路，并要求生产环境启用严格校验开关。
2. 业务域：User、Team、Tournament、Match、Schedule、Message、Media 等 MVP 域已落地。
3. 运行保障：已形成发布回滚、成本审计与 stg 环境开关脚本。

## 4. 约束与边界
1. 接口字段、错误码、时间格式以 `docs/03_接口与数据契约/` 为唯一事实源。
2. 端内实现改动必须在各自子仓完成；跨端规则改动必须先回写根仓文档。
3. 任何跨端行为变化都需要先更新契约或规范，再推进实现与联调。

## 5. 风险与待办
1. iOS 提审流程仍受“技术支持 URL 固化”进度影响。
2. Flutter 端需补齐与 iOS 同构的完整自动化验收链路。
3. 云资源仍有持续优化空间（重点在 stg 关闭策略与长期成本控制）。

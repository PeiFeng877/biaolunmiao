# 当前系统架构基线

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v1.4
**日期**: 2026-03-22
**适用范围**: `bianlunmiao-ios/`、`bianlunmiao-admin/`、`辩论喵-后端/`、`docs/04_测试与发布/`

## 1. 当前系统形态
1. 架构模式：客户端 + 服务端 API 的前后端分离模式。
2. 现行目标后端：阿里云 `FC Web 函数 + FastAPI`。
3. 对外契约：统一收口为 `GET /healthz` 与 `POST /api` 的单入口 RPC。
4. 环境策略：仅保留 `local` 与 `prod` 两态，当前不保留云端测试服。

## 2. 端侧现状
1. iOS：为当前主交付端，通过数据层 RPC 映射兼容既有页面与状态模型。
2. Web：管理端纳入本轮正式发布范围，作为正式运营工作台与后端管理入口。
3. 根仓 `docs/`：作为跨端 SSOT，统一维护产品、架构、契约、测试与发布规范。

## 3. 后端现状
1. 鉴权：App 用户会话与后台管理员会话分离，分别由 `sessions` 与 `admin_sessions` 承载。
2. 业务域：User、Team、Tournament、Match、Schedule、Message、Media 由 FastAPI 统一提供。
3. 运行保障：现行目标资源为 `FC + 新购 RDS PostgreSQL Serverless + OSS`，旧 SAE/EMAS/CLB/ECS 仅保留历史记录。

## 4. 约束与边界
1. 接口字段、错误码、时间格式以 `docs/03_接口与数据契约/` 为唯一事实源。
2. 端内实现改动按 `bianlunmiao-ios/` 与 `辩论喵-后端/` 目录治理；跨端规则改动必须先回写根仓文档。
3. 任何跨端行为变化都需要先更新契约或规范，再推进实现与联调。

## 5. 风险与待办
1. 旧资源与新资源的切换顺序必须先完成本地联调，再部署 FC 默认域名，再接正式自定义域名。
2. 新购 RDS PostgreSQL Serverless 的 `AutoPause`、最小规格与种子初始化必须先验证。
3. TestFlight 与正式后端共用一套数据时，必须强制使用专用测试账号与命名规范。

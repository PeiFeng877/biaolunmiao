## 1. 实体数据定义 (Entity Schema)

### 1.1 队伍 (Team)
*   **UUID (Internal)**: 系统后台生成的 32 位唯一标识符，用于数据库关联，**用户不可见**。
*   **Public ID (Display)**: 系统自动分配的短 ID（如 6-8 位数字），**唯一，用于搜索和展示**。
*   **名称 (Name)**: 允许重复（如可能有 10 个“复仇者队”），依靠 Public ID 区分。
*   **简介 (Bio)**: 文本描述。
*   **头像 (Avatar)**: 图片 URL。

### 1.2 用户 (User)
*   **UUID (Internal)**: 系统后台唯一标识。
*   **User ID (Custom)**: **用户自定义的唯一标识**（字母+数字组合），用于搜索和精准定位。
*   **昵称 (Nickname)**: 允许重复。

## 2. 交互流程：创建队伍

### 2.1 入口
- **位置**: App 底部 Tab 栏 -> **“队伍”**（Tab 1）。
- **交互**: 右上角 **“+”** -> 选择“创建队伍”。

### 2.2 创建界面
- **形式**: 模态弹窗（Modal Sheet）。
- **字段**:
  - 头像（选填）
  - 队伍名称（必填）
  - 队伍码 / Public ID（系统生成，只读，用于加入）
  - 简介（选填）

### 2.3 后置流程 (Post-Action)
*   窗口关闭后，自动跳转至新创建队伍的 **“队伍详情/管理页”**。

---

## 3. 交互流程：成员管理

### 3.1 邀请加入（可选，MVP 1.0 可后置）
- MVP 1.0 优先支持“申请加入 + 审批”闭环；邀请加入可作为增强项。

### 3.2 申请加入（主动方：申请人）
- **入口**: 队伍 Tab -> “加入队伍” -> 输入队伍码（Public ID）或搜索队伍。
- **操作**: 提交申请。
- **结果**: 进入“待审批”状态。

### 3.3 审批流程（主动方：队长/管理员）
- **入口**: 队伍详情页 -> “待处理申请”。
- **操作**:
  - 同意：用户正式入队
  - 拒绝：申请单归档

## 4. 权限管理逻辑 (Permission Matrix)

| 权限操作 | 队长 (Owner) | 管理员 (Admin) | 队员 (Member) |
| :--- | :---: | :---: | :---: |
| **创建队伍** | ✅ | ✅ | ✅ |
| **解散队伍** | ✅ | ❌ | ❌ |
| **转让队长** | ✅ | ❌ | ❌ |
| **指派/撤销管理员** | ✅ | ❌ | ❌ |
| **邀请成员** | ✅ | ✅ | ❌ |
| **审批入队申请** | ✅ | ✅ | ❌ |
| **移除成员 (踢人)** | ✅ (含管理) | ✅ (仅队员) | ❌ |
| **编辑队伍资料** | ✅ | ✅ | ❌ |

*注：一个人可以是 A 队的队长，同时是 B 队的队员。权限是绑定在“人-队”关系上的，而非全局属性。*

## 5. 页面结构概览

1. **队伍列表页**（Tab 1）
   - 展示我加入的所有队伍，并标记“队长/管理员/队员”身份
2. **队伍详情页**
   - 头部：头像、名称、队伍码、简介（可编辑：队长/管理员）
   - 待办：入队申请（仅队长/管理员）
   - 成员列表：按角色标签展示；可进入队员详情
3. **队员详情页**
   - 履历/战绩（轻量）
   - 赛程：未来 + 历史

# 单仓发版执行规范

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v1.5
**日期**: 2026-03-20
**适用范围**: 主仓 `main` 分支下的 iOS 提审、后端上线与联合发版

## 1. 目标
1. 固化单仓模式下的发版入口，避免“代码已改完，但不知道该怎么发”。
2. 让 Git 操作、测试门禁、iOS 提审资料、后端上线步骤都可重复执行。
3. 让产品、研发、后续协作者对每次发版有统一判断口径。

## 1.1 主文档定位
1. 本文件是当前现行唯一发版流程主文档。
2. `docs/04_测试与发布/01_规范/07_中国大陆App备案与AppStore上架一体化手册.md`、`08_中国大陆备案资料包_辩论喵_v1.md`、`09_中国大陆iOS上架备案执行计划.md` 仅作为合规与资料补充，不替代本流程。
3. `docs/04_测试与发布/02_记录/` 下的文档仅用于保存轮次化执行记录，不作为现行流程规范。

## 2. 发布类型
1. `iOS 提审`: 只发 iOS 新包，不要求同日改后端生产逻辑。
2. `后端上线`: 只发后端服务，不要求同日提交 iOS 新包。
3. `联合发版`: 同一需求同时改 iOS、后端、文档，并在同一轮发布中落地。

## 3. Git 标准流程
### 3.1 开发分支
1. 所有功能、修复、提审整改都从 `main` 切新分支。
2. 分支命名统一使用 `codex/<topic>`。

```bash
git checkout main
git pull --ff-only origin main
git checkout -b codex/<topic>
```

### 3.2 开发完成后的收口
1. 保证 `git status --short` 为空或只剩本次主题文件。
2. 若接口、数据模型、发布流程变化，必须同步更新 `docs/03_接口与数据契约/` 或 `docs/04_测试与发布/`。
3. 在功能分支先跑完本次需要的最小测试，再合回 `main`。

### 3.3 合回主干
1. 只允许把已经验收通过的功能分支合到 `main`。
2. 默认使用快进合并，避免把无关分支历史揉成一团。

```bash
git checkout main
git merge --ff-only codex/<topic>
```

### 3.4 发版打点
1. 正式发版前，`main` 必须指向本次最终代码。
2. iOS 和后端分别打各自可追溯 tag。
3. 发版完成后推送 `main` 与 tag。

建议 tag 规则：
1. iOS：`ios/v<版本号>-build<构建号>`
2. 后端：`backend/<YYYY-MM-DD>-<shortsha>`
3. 联合发版：可补一个总 tag，例如 `release/<YYYY-MM-DD>`

```bash
git tag ios/v1.0.0-build5
git tag backend/2026-03-14-78f5e65
git push origin main --tags
```

## 4. 公共发版门禁
1. `git status --short` 为空。
2. 需要回写的文档已经提交到主仓。
3. 关键命令结果可复述，不允许“我本地好像跑过”。
4. 正式发版前，在 `docs/04_测试与发布/02_记录/` 补一份本轮记录，写明版本、提交、测试结果、风险与回滚点。

## 5. iOS 发版流程
### 5.1 发版前必须完成
1. 更新版本号与构建号。
2. 运行治理检查：

```bash
swift bianlunmiao-ios/docs/03_Governance/tools/governance_audit.swift --mode check --root bianlunmiao-ios
```

3. 运行 iOS 单测：

```bash
xcodebuild test -project bianlunmiao-ios/BianLunMiao.xcodeproj -scheme BianLunMiao -destination 'platform=iOS Simulator,name=iPhone 17 Pro,OS=26.3.1' -only-testing:BianLunMiaoTests CODE_SIGNING_ALLOWED=NO
```

4. 若本轮改了 UI 或关键流程，再补一次 UI 冒烟：

```bash
bash scripts/ios_ui_lane.sh smoke-local
```

5. 补一次真机人工验收，至少覆盖：
   - Apple 登录
   - 首次启动
   - 队伍/赛事/日程/消息关键路径
   - Release 环境网络连通性

### 5.2 提审前必须更新的信息
1. App Store Connect 中的版本号、构建号。
2. 隐私政策 URL。
3. 技术支持 URL。
4. 截图与 1024 图标。
5. 年龄分级、隐私问卷、审核备注。
6. 如涉及中国大陆上架：备案编号、供应信息、App 内备案展示位。
7. 如审核需要复现路径：提供测试账号、操作步骤、审核备注说明。

### 5.3 构建与上传
1. 产物必须来自干净工作区的 `main`。
2. 归档、导出、上传沿用当前工程命令链：

```bash
xcodebuild -project bianlunmiao-ios/BianLunMiao.xcodeproj -scheme BianLunMiao -configuration Release -archivePath build/release/BianLunMiao.xcarchive archive
xcodebuild -exportArchive -archivePath build/release/BianLunMiao.xcarchive -exportOptionsPlist bianlunmiao-ios/build/release/ExportOptions-appstore-connect.plist -exportPath build/release/export -allowProvisioningUpdates
asc builds upload --app 6759727855 --ipa build/release/export/BianLunMiao.ipa --wait
```

3. 上传后必须记录：
   - Git commit
   - 版本号与构建号
   - Archive 路径
   - IPA 路径
   - App Store Connect Build ID

## 6. 后端发版流程
### 6.1 发版前必须完成
1. 若本轮改了接口或数据模型，先更新 `docs/03_接口与数据契约/`。
2. 若本轮改的是后端，运行后端代码检查：

```bash
make lint
```

3. 若本轮改的是后端，运行后端发布前测试：

```bash
uv run --extra dev python -m pytest tests/test_mvp_api.py -q
```

4. 若本轮改的是正式后端，先在本地完成 `GET /healthz` 与 `POST /api` 冒烟，再准备 FC 默认域名部署。

5. 迁移阶段说明：
   - 现行目标后端为 `FC Web 函数 + FastAPI`。
   - 历史 `辩论喵-后端-serverless/` 与 `EMAS` 相关实现仅保留为历史参考，不再作为现行发版入口。
   - 正式环境先使用 FC 默认域名承接流量，`api.bianlunmiao.top` 本轮不切。

### 6.2 正式环境流程
1. 先完成本机联调和最小回归，再部署正式后端。

2. 正式环境先用 FC 默认域名，至少覆盖：
   - `GET /healthz -> 200`
   - `auth.debug_token -> 403 DEBUG_TOKEN_DISABLED`
   - 真机 Apple 登录成功并进入空数据主容器
   - 头像上传成功且 `OSS` 可见文件
3. 若有数据结构调整，先确认当前是否允许“空库重建”；进入真实流量阶段后再补充细化回滚策略。
4. 发布完成后，立即检查正式入口、关键接口响应、应用日志与错误率。

### 6.3 必须记录的信息
1. Git commit 与 tag。
2. 发布环境：`prod`。
3. 是否执行数据库迁移。
4. 健康检查结果。
5. 回滚点。

## 7. 联合发版额外要求
1. iOS 与后端同时改动时，先确认 `docs/03_接口与数据契约/` 已同步。
2. 联合发版默认顺序：
   - 后端本地联调
   - iOS 联调验证
   - 后端正式环境
   - iOS 正式 Archive / 上传 / 提审
3. 禁止先提审 iOS，再临时补后端破坏性改动。

## 8. 回滚规则
### 8.1 iOS
1. 已上传的错误构建不回写覆盖，只能出新 build。
2. 若提审后发现问题，基于 `main` 新建修复分支，修完后重新归档上传。

### 8.2 后端
1. 应用层优先回滚到上一个稳定版本。
2. 数据层优先执行已验证的 `downgrade`；若不可逆，必须先恢复服务再补救数据。
3. 出现 P0/P1 事故时，先止血，再补复盘记录。

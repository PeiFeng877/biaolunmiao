# iOS 与后端自动化测试现状审计

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v1.0
**日期**: 2026-03-14
**适用范围**: `bianlunmiao-ios/`、`辩论喵-后端/`

## 1. 背景与范围
本次审计只看两件事：
1. iOS 与后端的自动化测试现在做到了什么程度。
2. 这些测试在当前仓库里是否能稳定执行、是否足以支撑提测与联调。

本次不改业务逻辑，不承诺顺手修复测试问题；目标是把现状讲清楚，方便产品、研发一起判断下一步优先级。

## 2. 一页结论
### 2.1 总览
| 模块 | 结论 | 颜色 | 业务影响 |
| --- | --- | --- | --- |
| iOS 单元测试 | 已形成可用主干，当前实测可通过 | 绿 | 对日常改动的逻辑回归有直接保护 |
| iOS UI 自动化 | 已有多条流程脚本，能跑通冒烟，但执行入口依赖具体模拟器型号 | 黄 | 影响提测稳定性，不影响现网功能 |
| iOS 治理检查 | 当前失败 | 红 | 说明文档清单与实际实现不同步，影响交接与审查效率 |
| 后端 lint | 当前通过 | 绿 | 代码风格与基础静态检查可用 |
| 后端 pytest 用例本体 | 当前可通过 | 绿 | 核心接口回归具备自动化基础 |
| 后端 `make test` 入口 | 当前失败 | 红 | 团队成员按文档执行时会直接卡住，影响联调与提测节奏 |

### 2.2 管理层结论
1. iOS 自动化测试的“内容完成度”明显高于“执行入口稳定性”问题。
2. 后端自动化测试的“用例完成度”并不差，真正的短板是门禁命令没有整理好。
3. 当前最需要优先修的不是“补更多测试”，而是把现有测试入口修到拿来即跑。

## 3. iOS 自动化测试现状
### 3.1 iOS 单元测试
| 项目 | 内容 |
| --- | --- |
| 现状 | `BianLunMiaoTests` 已覆盖队伍申请、消息聚合、日程来源、资料编辑、赛事/场次/阵容/赛果、设计治理等核心逻辑。 |
| 证据 | 在 `iPhone 17 Pro` 模拟器上执行 `xcodebuild test -project BianLunMiao.xcodeproj -scheme BianLunMiao -destination 'platform=iOS Simulator,name=iPhone 17 Pro,OS=26.3.1' -only-testing:BianLunMiaoTests`，结果为 `TEST SUCCEEDED`。 |
| 影响 | 这部分已经能为主流程逻辑改动提供回归保护，不是“空壳测试”。 |
| 建议 | 保持现有覆盖方向，下一步优先把测试命令写进稳定脚本或统一文档，减少环境依赖。 |

### 3.2 iOS UI 自动化
| 项目 | 内容 |
| --- | --- |
| 现状 | 已存在登录页、远端联调、队伍、赛事、消息、我的、日程、截图包等多条 UI 测试脚本。 |
| 证据 | `BianLunMiaoUITests.swift` 中已定义多条 `test...` 用例；执行 `xcodebuild test ... -only-testing:BianLunMiaoUITests/BianLunMiaoUITests/testExample` 可实际通过。 |
| 影响 | 说明 UI 自动化不是摆设，至少具备冒烟验证能力。 |
| 建议 | 保留当前覆盖面，优先解决执行入口依赖具体设备名的问题，再决定是否扩大用例数量。 |

### 3.3 iOS UI 自动化的主要短板
| 项目 | 内容 |
| --- | --- |
| 现状 | 按旧命令执行时，指定的 `iPhone 16` 模拟器在当前机器上不存在，导致命令直接失败。 |
| 证据 | `xcodebuild` 明确报错：无法找到 `platform=iOS Simulator,name=iPhone 16` 对应设备。 |
| 影响 | 同一套测试在不同机器上不一定拿来即跑，影响提测稳定性与团队协作效率。 |
| 建议 | 将测试入口改为读取本机可用设备，或统一为当前团队标准模拟器型号。 |

### 3.4 iOS 治理检查
| 项目 | 内容 |
| --- | --- |
| 现状 | 当前失败。 |
| 证据 | 执行 `swift docs/03_Governance/tools/governance_audit.swift --mode check` 后，提示 `按钮使用清单.md` 与 `反馈使用清单.md` 已过期。 |
| 影响 | 不直接影响线上功能，但会让“规范文档是否反映真实实现”这件事失真。 |
| 建议 | 先重新生成 inventory，再决定是否把治理检查提升为更强门禁。 |

## 4. 后端自动化测试现状
### 4.1 后端 lint
| 项目 | 内容 |
| --- | --- |
| 现状 | 当前通过。 |
| 证据 | 执行 `make lint`，结果为 `uv run ruff check .` 且 `All checks passed!`。 |
| 影响 | 基础静态检查可用，说明最外层代码质量门槛已建立。 |
| 建议 | 保持不变，后续只需并入统一门禁流水线。 |

### 4.2 后端 pytest 用例本体
| 项目 | 内容 |
| --- | --- |
| 现状 | 当前可通过，且覆盖的是关键接口语义。 |
| 证据 | 执行 `uv run --extra dev python -m pytest tests/test_mvp_api.py -q`，20 条用例全部通过。 |
| 影响 | 登录、Apple token 校验、JWKS 缓存、入队申请、消息确认、赛事列表、账号删除、媒体上传 token 等核心接口已有自动化保护。 |
| 建议 | 后续优先补齐测试入口与依赖声明，不必先大规模重写测试内容。 |

### 4.3 后端 `make test` 入口
| 项目 | 内容 |
| --- | --- |
| 现状 | 当前失败。 |
| 证据 | 执行 `make test` 时，`uv run pytest` 报错 `Failed to spawn: pytest`。 |
| 影响 | 团队成员按当前文档或 Makefile 执行测试会直接卡住，影响联调效率，也会让“测试是否可执行”产生误判。 |
| 建议 | 优先修复 Makefile 与依赖安装入口，让默认命令就能跑通现有测试。 |

## 5. 其他规范问题
### 5.1 文档与真实仓库状态不一致
| 项目 | 内容 |
| --- | --- |
| 现状 | 删除 Flutter 前，现行文档长期把 Flutter 写成当前执行范围，但仓库实际没有可运行 Flutter 工程。 |
| 证据 | 现已将 Flutter 相关现行文档迁入历史归档，并从 SSOT 中移除。 |
| 影响 | 会让产品、研发对当前投入方向产生误解。 |
| 建议 | 继续保持“现行文档只写当前真正维护的范围”。 |

### 5.2 长文件与目录超限
| 项目 | 内容 |
| --- | --- |
| 现状 | iOS 存在超过 800 行的大文件；部分文档目录曾长期堆积过多文件。 |
| 证据 | `AppStore.swift`、`RemoteGateway.swift`、`TournamentDetailView.swift` 均超过建议阈值。 |
| 影响 | 后续迭代和代码评审成本会上升，但不属于当前线上阻塞。 |
| 建议 | 作为下一轮结构治理任务拆分处理，不与本轮测试审计混在一起。 |

### 5.3 子仓 README 文档基线不统一
| 项目 | 内容 |
| --- | --- |
| 现状 | 部分子仓 README 仍未完全满足根仓要求的协议头、版本、日期、变更日志规范。 |
| 证据 | 后端 `README.md` 当前仍缺少根仓要求的完整文档协议头信息。 |
| 影响 | 影响新成员理解项目，也削弱“文档即协作入口”的一致性。 |
| 建议 | 作为文档治理任务单独补齐，不在本轮与业务测试整改混做。 |

## 6. 建议优先级
### P1：先解决“测试入口能不能跑”
1. 修后端 `make test`，让默认命令直接可执行。
2. 统一 iOS UI 自动化的模拟器目标，避免设备名不匹配导致假失败。

### P2：再解决“规范和实现是否一致”
1. 更新 iOS 治理 inventory 文档，让治理检查恢复通过。
2. 补齐后端 README 的协议头与变更日志。

### P3：最后处理结构优化
1. 拆分超长 iOS 文件。
2. 持续清理文档目录层级，避免再次堆积。

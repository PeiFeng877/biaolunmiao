## 0. 适用范围

1. 本文件同时服务 App 业务数据模型与 Web 管理后台所需的数据模型。
2. 管理端前端通过 `/api/v1/admin/*` 消费这里定义的 `admin_*` 表语义，不复用 App 用户鉴权模型。
3. 若后台业务扩展到新的管理域，优先在本文件补充数据表与约束，再推进前端页面。

## 3. 关键约束

1. `team_join_requests`：
   - 同一申请人对同一队伍仅允许 1 条 `pending` 记录。
   - 已通过/已拒绝不影响再次申请。

2. `match_rosters`：
   - 唯一约束 `(match_id, team_id, position)`。
   - 保证同一队伍同一场次不会重复辩位。

3. 主键与公开标识：
   - 主键统一 UUID。
   - 用户/队伍对外使用 `public_id`。

4. `users` 账号删除：
   - 使用 `status = 1 (DELETED)` 表示账号已删除。
   - 新增 `deleted_at` 记录删除时间。
   - 删除时清空 `apple_sub` 绑定，允许同一 Apple 账号重新注册新账号。
   - 软删除后保留最小必要审计数据，不恢复旧账号、不迁移旧数据。

5. `refresh_tokens` 会话撤销：
   - 账号删除时必须批量写入 `revoked_at`。
   - 删除账号后刷新会话请求必须失效。

6. `admin_users` 管理员体系：
   - 与 `users` 业务用户体系完全隔离。
   - 使用独立 `email`、`password_hash`、`role`、`status` 字段。
   - 首期角色固定为 `super_admin` 与 `editor`。

7. `admin_refresh_tokens` 后台会话：
   - 不与业务 `refresh_tokens` 复用。
   - 仅服务 `/api/v1/admin/*` 会话续签与主动退出。

8. `admin_audit_logs` 审计日志：
   - 所有后台写操作必须写入。
   - 至少保存 `actor_admin_id`、`resource_type`、`resource_id`、`action`、`before_json`、`after_json`、`created_at`。

## 4. 迁移策略

1. 第一步：建基础表 + 外键 + 唯一索引。
2. 第二步：写入初始种子（调试用户、样例队伍）。
3. 第三步：iOS 开始联调，优先走 Team 闭环。
4. 第四步：根据联调反馈新增变更迁移（可回滚）。

## 5. 回滚策略

- 每个迁移都必须提供 `downgrade`。
- 严禁在未备份情况下直接改线上结构。
- 高风险变更先在 `staging` 验证。

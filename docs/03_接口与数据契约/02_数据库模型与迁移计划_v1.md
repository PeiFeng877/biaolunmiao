# 数据库模型与迁移计划 v2

**版本**: v2.0
**日期**: 2026-03-22

## 0. 适用范围

1. 本文件同时服务 App 业务数据模型与 `TestFlight` 共用的正式后端数据模型。
2. 现行目标态为新购杭州 `RDS PostgreSQL Serverless` 的关系型表模型，不再采用 EMAS 集合模型。
3. Web 管理后台本轮不进入现行范围，`admin_*` 模型仅作为未来扩展预留。
4. 历史 Python 后端表结构仅作为参考，本期不迁移旧数据。

## 1. 现行目标表

- `users`
- `sessions`
- `teams`
- `team_members`
- `join_requests`
- `tournaments`
- `matches`
- `match_rosters`
- `messages`
- `schedule_sources`
- `media_files`
- `counters`

## 2. 兼容参考模型

- 历史 Python 后端中的 `refresh_tokens`、`team_join_requests`、`tournament_participants`、`user_message_status` 仍可作为迁移参考。
- `admin_users`、`admin_sessions` 仅作为未来恢复管理端时的预留结构，不作为当前正式环境必需项。

## 3. 关键约束

1. `join_requests`：
   - 同一申请人对同一队伍仅允许 1 条 `pending` 记录。
   - 已通过/已拒绝不影响再次申请。

2. `match_rosters`：
   - 同一 `match_id + team_id + position` 只允许一条有效记录。
   - 保证同一队伍同一场次不会重复辩位。

3. 主键与公开标识：
   - 主键统一 UUID 或等价全局唯一标识。
   - 用户/队伍对外使用 `public_id`。

4. `users` 账号删除：
   - 使用 `status = 1 (DELETED)` 表示账号已删除。
   - 新增 `deleted_at` 记录删除时间。
   - 删除时清空 `apple_sub` 绑定，允许同一 Apple 账号重新注册新账号。
   - 软删除后保留最小必要审计数据，不恢复旧账号、不迁移旧数据。

5. `sessions` 会话撤销：
   - 账号删除时必须批量写入 `revoked_at`。
   - 删除账号后刷新会话请求必须失效。

6. `messages`：
   - 接收人维度按 `created_at` 倒序读取。
   - `ack` 行为只更新目标消息的已读状态，不改写历史内容。

7. `counters`：
   - 负责生成用户/队伍等公开标识的顺序号。
   - 不要求历史数值迁移，允许迁移期从空库重新计数。

8. 测试账号与命名规范：
   - `TestFlight` 共享正式数据库，但必须使用专用测试账号、测试队伍和测试赛事命名规范。
   - 测试数据的公开展示名应一眼可识别用途，避免污染正式数据。

## 4. 迁移策略

1. 第一步：新购杭州 `RDS PostgreSQL Serverless`，开启 `AutoPause`，按最小配置创建空 schema 与索引。
2. 第二步：写入最小种子（调试用户、样例队伍、默认 schedule source）。
3. 第三步：iOS 开始联调，优先走 `healthz/auth/users/teams` 闭环。
4. 第四步：补齐 `tournaments/messages/schedule/media` 并在 `TestFlight` 上复用同一套正式后端。

## 5. 回滚策略

- 新库在正式切流前不承载历史数据，因此优先采用“重建资源 + 重放种子”的回滚方式。
- 旧 RDS 在新库可用前继续保留，便于必要时回退到旧后端读取路径。
- 若后续正式切流后需要细化回滚，再补充数据回滚方案。

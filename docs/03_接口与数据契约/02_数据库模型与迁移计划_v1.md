## 1. 复用核心表

- `users`
- `teams`
- `team_members`
- `tournaments`
- `matches`
- `match_rosters`

## 2. 新增必需表

- `refresh_tokens`
- `team_join_requests`
- `tournament_participants`
- `schedule_sources`
- `messages`
- `user_message_status`

## 3. 关键约束

1. `team_join_requests`：
   - 同一申请人对同一队伍仅允许 1 条 `pending` 记录。
   - 已通过/已拒绝不影响再次申请。

2. `match_rosters`：
   - 唯一约束 `(match_id, team_id, position)`。
   - 保证同一队伍同一场次不会重复辩位。

3. 主键与公开标识：
   - 主键统一 UUID。
   - 用户/队伍对外使用 `public_id`。

## 4. 迁移策略

1. 第一步：建基础表 + 外键 + 唯一索引。
2. 第二步：写入初始种子（调试用户、样例队伍）。
3. 第三步：iOS 开始联调，优先走 Team 闭环。
4. 第四步：根据联调反馈新增变更迁移（可回滚）。

## 5. 回滚策略

- 每个迁移都必须提供 `downgrade`。
- 严禁在未备份情况下直接改线上结构。
- 高风险变更先在 `staging` 验证。

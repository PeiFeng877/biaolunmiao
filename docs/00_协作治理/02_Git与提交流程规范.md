# Git与提交流程规范

[PROTOCOL]: 变更时更新此头部，然后检查 agents.md

**版本**: v1.4
**日期**: 2026-03-14

## 1. 分支规范
1. 分支命名统一使用 `codex/<topic>`。
2. 默认短分支开发，尽快合并。
3. 禁止在主干长期堆积未审查变更。

## 2. 提交规范
1. 提交格式：`feat|fix|docs|refactor: <scope> <summary>`。
2. 一次提交聚焦一个主题，避免混杂无关改动。
3. 涉及契约变更时，提交必须包含文档更新。

## 3. PR 规范
1. PR 描述必须包含：影响模块、接口变化、测试结果、回滚方案。
2. 跨端 PR 必须附带契约文档引用。
3. 门禁失败不得合并。

## 4. 发布前检查
1. iOS/后端最小测试通过。
2. 文档协议检查通过（版本、日期、变更日志）。
3. 关键路径支持回滚。

## 5. 单仓协作约束
1. 主仓统一跟踪 `docs/`、`bianlunmiao-ios/`、`辩论喵-后端/`、`scripts/` 与治理文件。
2. 同一功能可在一个提交中同时包含文档、iOS、后端改动，但必须属于同一业务主题。
3. 涉及跨端契约变化时，仍以 `docs/03_接口与数据契约/` 为先，再同步实现代码。
4. 单仓不再按历史上的仓库路由分流，而是按“本次功能是否聚焦”判断提交边界。

## 6. 提交范围判定（强制）
1. `docs/**`、`bianlunmiao-ios/**`、`辩论喵-后端/**` 可在同一提交中共存，但必须服务于同一功能或同一治理动作。
2. 若一次改动同时包含多个不相关主题，必须拆分为多个提交。
3. 发现与本次主题无关的文件出现在 `git status`，必须先剥离再提交。
4. 单仓内禁止用“目录不同”替代“职责清晰”；目录只是边界，不是提交理由。

## 7. 提交前检查清单（强制）
1. `pwd`：确认当前目录位于主仓工作区。
2. `git rev-parse --show-toplevel`：确认当前 Git 根。
3. `git status --short`：确认提交集与本次功能范围一致。
4. 若同时改动 `docs/`、iOS、后端：在提交说明中明确写出这三者的关联原因。
